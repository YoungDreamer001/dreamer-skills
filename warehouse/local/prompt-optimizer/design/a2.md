在 GPT-5.5 下，角色设定role play、few-shot、prompt chaining 这些prompt方法的地位降级了。

早先我进行复杂任务分析的时候，就发现在prompt中进行角色设定（你是xx领域的专家）不是那么有效，更重要的是通过prompt激发恰当的模型推理能力。角色只是一顶帽子，一层皮，带来的更多的是风格差异，而不是内在能力。就像现实中很多xx专家，实际上是骗子和搅屎棍一样。

在 GPT-5.5 下，role play、few-shot、prompt chaining 这些方法并没有完全失效。失效的是旧用法：把它们当成“控制模型智力”的主要手段，写一大段“你是世界顶级专家，请一步一步深入思考”的 prompt。不是“role play 没用了”，而是 role play 要降级为“任务身份设定”，更适合定义视角而不是能力。

GPT-5.5 prompting guide 仍然把 Role 放在复杂 prompt 的建议结构里，但只建议 1 到 2 句，说明模型的 function、context 和 job。它还明确区分 personality 和 collaboration style，并提醒两者都要短，不能替代目标、成功标准、工具规则和停止条件。

在更强的模型下，prompt要的是给模型的推理设置任务坐标系，这是激发模型正确的推理能力的关键。

GPT-5.5 本身已经有更强的规划、推理和执行能力。我们要做的不是用复杂 prompt “逼它思考”，而是用这些元素告诉它：往哪里推理，按什么标准判断，哪些边界不能越过，什么时候该停。官方 prompting guide 也明确说，GPT-5.5 最强的时候，是 prompt 定义目标结果、成功标准、约束和可用上下文，然后让模型自己选择路径；不要默认把每一步流程都写死。

1，目标决定推理方向。
它回答“你最终要解决什么问题”。没有目标，模型容易变成泛泛分析。比如“分析这篇论文”太宽。“判断这篇 AI 论文是否值得纳入 journal club 重点讨论”就明确很多。

2，Success criteria 决定判断标准。
它回答“什么才算做对”。这是最重要的部分之一。比如你要求模型评估 AI 论文，成功标准可以是：必须区分研究类型、数据集、验证方式、endpoint、适用性和主要局限。这样模型的推理会围绕这些维度展开，而不是只写漂亮总结。官方文档也把 success criteria 放在 outcome-first prompt 的核心位置。

3，Constraints 决定边界。
它回答“哪些事情不能做，哪些条件必须遵守”。比如不能编造数据，不能把 technical performance 说成 clinical benefit，不能把单中心回顾性研究说成广泛临床验证。constraints 不是越多越好。只写真正会改变质量或风险的限制。

4，Evidence rules 决定推理的落点。
它回答“哪些结论必须有证据，证据不足时怎么办”。这对医学、科研、产品判断特别关键。官方 guide 也说，grounded answers 里 citation behavior 应成为 prompt 的一部分，要定义哪些内容需要支持、什么算足够证据、证据缺失时模型如何处理；没有证据不应自动变成事实性否定。

5，Output 决定推理如何呈现。
它回答“最后要以什么形式交付”。输出格式会反过来影响模型组织信息的方式。比如你要求“结论、证据表、局限、适用性、下一步”，模型就更容易形成可审计结构。官方建议把输出形态、受众、长度和格式说清楚，但不要让格式比内容更重。

6，Stop rules 决定什么时候停止搜索、追问、迭代或拒答。
这是很多人忽略的部分。没有 stop rules，模型可能过度搜索、过度解释、反复补充，或者在证据不足时硬写结论。官方 guide 专门建议加入 stopping conditions，并给出“用最少但足够的证据正确回答，精确引用，然后停止”的思路。

以下为GPT 5.5官方说明

人格与行为
GPT-5.5 的默认风格高效、直接、以任务为导向。这对生产系统很有用：回答保持聚焦，行为更容易引导，模型也会避免不必要的对话填充。

对于面向客户的助手、支持工作流、教练式体验和其他对话型产品，你需要同时定义 personality 和 collaboration style。
• Personality 控制助手听起来是什么样：语气、温度、直接程度、正式程度、幽默感、共情程度和打磨程度。
• Collaboration style 控制助手如何工作：什么时候提问，什么时候做假设，主动性多强，给多少上下文，什么时候检查工作，以及如何处理不确定性或风险。

两者都要保持简短。Personality 指令应塑造用户体验。Collaboration 指令应塑造任务行为。它们都不能替代清楚的目标、成功标准、工具规则或停止条件。

稳健、以任务为中心的助手人格示例：
# Personality
你是一个可靠的协作者：亲切、稳定、直接。假设用户有能力，并且出于善意行动。用耐心、尊重和务实的帮助来回应。
 
当请求已经足够清楚，可以尝试推进时，优先推进，而不是停下来追问。使用上下文和合理假设继续前进。只有当缺失信息会实质性改变答案或造成明显风险时，才提出澄清问题，并且问题要窄。
 
保持简洁，但不要显得生硬。给用户足够的上下文，让他们理解并信任答案，然后停止。只有当例子、比较或简单类比能让观点更容易理解时才使用。纠正用户或表达不同意见时，要坦率但有建设性。当用户指出错误时，直接承认，并专注于修复。
 
在专业边界内匹配用户的语气。默认避免 emoji 和脏话，除非用户明确要求这种风格，或已经清楚地把这种风格设定为适合当前对话。

表达力更强、协作型助手的人格示例：
# Personality
采用鲜明的对话存在感：聪明、好奇，在合适时有一点玩味，并关注用户的思考。问题模糊时提出好问题。一旦上下文足够，就变得果断。
 
保持温暖、协作和精炼。对话应感觉轻松而有生命力，但不要为了聊天而聊天。提出真正的观点，而不是仅仅复述用户，同时始终回应用户的目标和约束。
 
当任务需要综合或建议时，要思考充分并有依据。上下文足够时，给出清晰建议，解释重要取舍，并指出不确定性，但不要显得回避。

对于表达力更强的产品，可以明确加入温暖、好奇、幽默或观点，但仍要保持简短。用 personality 塑造体验，不要用它弥补目标不清或任务说明缺失。
